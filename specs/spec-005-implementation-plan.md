# Spec 005：实施计划与迭代拆解

## 1. 目标
把当前已经确认的需求和架构，拆成可以直接执行的开发阶段，减少 Codex 一次性生成大而散代码的风险。

该 spec 的目标是回答三件事：
1. 先做什么
2. 每一阶段交付什么
3. 每一阶段怎么验收

---

## 2. 开发原则
1. **先主流程，后外围能力**
   - 优先打通登录 → 书架 → 阅读 → 生词本主链路

2. **先静态资源，后动态能力**
   - 第一版内容内置在包里，先把阅读体验做稳

3. **先本地闭环，后远程扩展**
   - 先本地数据稳定，再考虑正式短信和云同步

4. **每个阶段都应可运行、可验证**
   - 不接受只堆代码不形成可运行状态

---

## 3. 开发阶段建议

## Phase 0：工程初始化
### 目标
搭好可开发骨架。

### 包含内容
- 新建 Android Compose 项目
- 建立基础包结构
- 接入导航
- 接入 DataStore
- 接入 Room
- 建立主题与基础组件

### 交付物
- 可运行空应用
- 基础目录结构
- 空白路由可跳转

### 验收标准
- App 能启动
- 页面可导航
- Room / DataStore 可正常初始化

---

## Phase 1：账号与邀请注册
### 对应 spec
- `spec-001-auth-and-invite.md`

### 目标
打通用户进入主流程的入口。

### 包含内容
- 登录页
- 手机号输入与校验
- 固定验证码 `123456`
- 新用户邀请码流程
- 昵称补填页
- 登录态持久化
- 启动页自动跳转

### 交付物
- 可完成注册与登录的流程
- 重启后自动恢复登录态

### 验收标准
- 新用户可通过邀请码注册
- 老用户可直接登录
- 退出登录后能回到登录页

---

## Phase 2：内容资源接入
### 对应 spec
- `spec-002-content-bundle-and-import.md`

### 目标
让 App 能识别并展示远程书架资源，并支持本地已安装内容包阅读。

### 包含内容
- 约定 `content/books/` 作为内容源目录
- 发布远程 `bookshelf.json / catalog.json / package.zip`
- 解析本地安装包中的 `book.json / pages.json / words.json`
- 构建远程书架预览与本地阅读索引

### 交付物
- 可发布的样例绘本内容包
- 远程书架与本地安装包读取逻辑

### 验收标准
- App 启动后能看到远程书架列表数据
- 点开未安装书籍时能按需下载并打开
- 打开某本书时能读取页面与单词数据

---

## Phase 3：书架首页
### 对应 spec

### 目标

### 包含内容

- 双列网格绘本卡片

- 点击绘本卡片进入阅读页

### 验收标准

- 点击卡片可进入阅读页

---

## Phase 4：阅读器主流程
### 对应 spec

- `spec-004-app-architecture.md`

### 目标

### 验收标准

### 交付物

---

## Phase 6：生词本
### 目标
完成学习闭环的最小版本。

### 包含内容
- 生词收藏
- 生词全局去重
- 生词本列表页
- 删除二次确认
- 从阅读页右上角进入生词本

### 交付物
- 可用生词本页面
- 本地存储生词数据

### 验收标准

---

## Phase 7：内测打磨
### 目标
把 MVP 收到可以发内测包的程度。

### 包含内容
- UI 细节修整
- 文案统一
- 异常状态处理
- 首批 3-5 本内容接入
- 基础测试
- 打包内测版

### 交付物
- 可分发的 Android 内测包

### 验收标准
- 主流程无阻塞 Bug
- 资源能稳定加载
- 关键页面不会明显卡顿或崩溃

---

## 4. 推荐任务拆分粒度
给 Codex 下任务时，不要说“做完整 App”。
应按以下粒度拆：

### 合理粒度示例
- 实现登录页 Compose UI
- 实现 AuthViewModel 与状态流
- 实现 DataStore 登录态存储
- 实现 books 资源加载器
- 实现书架双列网格 UI
- 实现 ReaderScreen 翻页逻辑
- 实现生词本 Room 表与 DAO

### 不合理粒度示例
- 一次性做完整阅读 App
- 一次性生成全部页面和数据库和接口

---

## 5. 每阶段建议产出
每个阶段都建议固定产出：
1. 代码
2. 变更说明
3. 验证步骤
4. 已知问题
5. 下一阶段建议

这会让你后面和 Codex 配合更稳。

---

## 6. 风险控制
### 6.1 最大风险
不是代码本身，而是以下 3 件事：
1. 内容整理比预期更慢
2. 单词索引不准确导致点词体验差
3. 一次让 Codex生成过大范围代码导致结构失控

### 6.2 控制策略
- 每次只做一个 spec 或一个子任务
- 先跑通样例书，不先铺 3-5 本全量内容
- 每阶段都必须可运行
- 任何需求不明确时先提问再改代码

---

## 7. 推荐里程碑
### Milestone 1
- 完成账号流程
- 完成样例书资源加载

### Milestone 2
- 完成书架首页
- 完成阅读器翻页与进度保存

### Milestone 3
- 完成点词、点句、生词本
- 形成完整 MVP 主流程

### Milestone 4
- 接入首批 3-5 本内容
- 输出内测包

---

## 8. 验收总表
### MVP 必须完成
- 登录 / 注册
- 书架首页
- 阅读器
- 点词发音
- 点句播放
- 生词本
- 阅读进度保存
- 内置内容加载

### MVP 明确不做
- 跟读
- 家长端
- 正式短信服务
- 内容在线分发
- 复杂学习分析

---

## 9. 建议的下一步动作
如果接下来要进入实现阶段，我建议顺序是：

1. 建工程骨架
2. 做 `spec-001`
3. 做一套最小样例内容
4. 做 `spec-003` 里的书架 + 阅读器主链路
5. 最后补生词本

这是最稳的路径。
